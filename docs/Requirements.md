# 要件定義

## 目的

FANZAの商品を取得・選定し、利用可能なサンプル動画付きのPR投稿をXへ自動投稿する。親投稿への自分自身の返信として、FANZAアフィリエイトID付き商品URLを投稿する。候補不足時に件数合わせはしない。

## 管理画面GUI刷新・投稿内容履歴（Step 7A）

- 管理画面は左固定サイドバーと選択ページだけを表示する構成へ刷新する。ダッシュボード、商品管理、女優管理、投稿予定、投稿履歴、同期・実行、設定を分離し、旧縦長一括表示を廃止する。
- 実投稿の親投稿・返信投稿の本文は投稿履歴へ保存し、Basic認証下の管理APIで安全に一覧・詳細を閲覧できる。dryRunは従来どおり投稿履歴へ保存しない。

## 業務・投稿要件

- 日本時間の日次枠は、セール商品2本、指定女優の商品2本、お気に入り商品のうちセール中の商品1本、合計最大5本。
- 指定女優の初期値は北岡果林、依本しおり。管理画面から複数登録・管理できる。
- 親投稿は【PR】、商品名、商品紹介文、カテゴリに応じた確認済み情報、返信欄への誘導、利用可能な動画または画像を含める。
- 返信は作品案内文とFANZAアフィリエイトID付き商品URLを、親投稿への返信として投稿する。
- 同一商品IDは投稿後30日間、再投稿禁止とする。

## 機能要件

- DMM/FANZAの許可された方法・公開情報から、商品、販売状態、価格、セール・新作情報、サンプル動画の利用可否、アフィリエイトURLを取得・正規化する。
- 価格は任意項目とする。固定価格として安全に解釈できる整数だけを保存し、範囲表現、波ダッシュ付き、欠損、不明形式は`NULL`として商品自体は保存する。通常価格・販売価格の両方が固定価格で通常価格が高い場合だけセール商品とし、価格不明の商品はセール商品として扱わない。
- Chrome拡張機能は、利用者がFANZAお気に入り画面上でボタンを押した場合だけ動作する。公式商品URLを重複除去して最大20件抽出し、件数と安全な検証結果だけを表示して認証済みバックエンドAPIへ同期する。FANZAのCookie、ID、パスワード、localStorage、ページHTMLは読み出し・送信・保存しない。
- Chrome拡張機能のお気に入り同期はcheck-onlyを既定とし、その成功結果を同じ画面で確認した後の明示操作でのみ`persist=true`を送信する。対象外ページ、20件超過、安全に検証できないURL、VR作品、API失敗ではpersistを許可せず、バックグラウンド巡回や定期実行を持たない。
- Chrome拡張が同期対象として扱う商品リンクは、現行`video.dmm.co.jp/av/content`または旧`www.dmm.co.jp/digital/videoa/.../detail`のAV商品詳細形式に限定する。他floor・商品種別の公式リンクはAVへ推測変換せず、URL全文を出さない理由別件数として除外する。同一content_idの複数URL形式は1商品へ正規化する。
- お気に入り同期APIは一度に最大20件の公式FANZA/DMM商品URLだけを受け付け、URLから抽出したcontent_idを既存商品と照合する。check-onlyを既定とし、明示persist時も無効URL、metadata取得不能、VR作品または保存失敗が含まれる場合はfavoritesを変更しない。商品URL、認証情報、Cookieは同期結果やログへ出力しない。
- `FavoriteProductProvider`は同期された公式商品URLを起点にcontent_idを重複排除し、DMM WebサービスのItemListで商品metadataを補完する。価格は任意であり、VR作品は共通判定で除外し、個別取得失敗で他の商品確認を中断しない。
- metadata取得不能は`api_not_listed`、`id_mismatch`、`invalid_metadata`、`vr_excluded`、通信失敗へ安全に分類する。取得不能または除外が1件でもある集合を部分persistせず、URL・content_id・商品名を結果やログへ出さない。
- PostgreSQLを商品、お気に入り、指定女優、投稿履歴、日次実績、実行結果の正本とする。
- 投稿エンジンはセール、指定女優、お気に入りセールの3区分で候補を選定し、区分別と合計の上限を適用する。
- 親投稿の成功後にだけ、親投稿IDへ返信を投稿する。親投稿・返信の成功または失敗を記録する。
- 管理画面で指定女優、設定、投稿履歴、日次実績、同期結果、除外・失敗理由を確認・管理できる。
- 指定女優管理画面では、女優名・別名・有効状態・優先度・新作／セール対象・最低投稿間隔・週間上限を追加・編集・検索・絞込み・削除できる。削除前には確認を行い、通常は無効化を案内する。
- 投稿エンジンと分析エンジンは同一アプリ内でモジュール分離する。分析は初期版で実行・投稿・除外・失敗を集計する。
- `DRY_RUN` は実投稿をせず、実投稿と同じ判定経路で予定と結果を確認できるよう維持する。
- 商品タイトルに明記された割引、半額、ポイント還元、キャンペーン、売れ筋等を、価格を補助情報としてルールベースで解析し、事実ベースの投稿訴求候補へ渡せるようにする。AIによる投稿コピー生成とX投稿は後続とする。
- 投稿冒頭向けの短いキラーメッセージは、タイトル解析済みの事実だけから再現可能なルールで生成する。誇張、作品内容の推測、成人向けの具体的描写、AI API、投稿テンプレート全体、X投稿は含めない。
- 親投稿本文はPR表記、解析済みのキラーメッセージ、女優名、事実ベースの短い案内、セール情報、最大2個のハッシュタグから生成し、URLは含めない。URLは後続の返信投稿で扱う。
- 親投稿の成功後、同一投稿への返信としてHTTP/HTTPSのアフィリエイトURLを1回だけ投稿できるようにする。dryRunでは通信せず、親投稿成功後の返信失敗は親投稿IDを含む部分成功として扱う。
- 親投稿には利用可能なサンプル動画を優先して1件添付し、動画が取得不能・非media応答の場合は公式商品画像1件へ安全にfallbackする。動画・画像のどちらも検証できない候補は投稿しない。dry-runは同じmedia選択・形式検証を通すがXへのuploadと投稿は行わない。
- 実X API投稿は`DRY_RUN=false`が明示された場合だけ行い、既定は安全なdryRunとする。実投稿履歴はPostgreSQLへ保存し、dryRunは実投稿済み判定へ混ぜない。
- 手動投稿要求は適格性確認を経由し、pending返信を優先して再試行する。30日再投稿禁止中はX APIを呼ばない。
- 当日候補はセール2、指定女優2、お気に入りセール1、合計最大5件を決定的に選定し、条件不足時は水増ししない。

## 自動投稿の必須条件

1. 販売中である。
2. アフィリエイトURLが正常で、アフィリエイトIDがある。
3. 親投稿に【PR】がある。
4. カテゴリ根拠が確認できる。セール系は正規な掲載確認、女優系は登録女優との厳密な関連を必要とする。価格は必須ではない。
5. 利用可能なサンプル動画または公式商品画像がある。
6. 過去30日以内に同一商品を投稿していない。
7. 投稿文が重複していない。
8. 区分別および日次の投稿上限以内である。

## 非機能要件

- フロントはHTML / CSS / JavaScript、バックエンドはNode.js / TypeScript、DBはPostgreSQLとする。
- Node.js常駐プロセスが定期実行し、X APIはOAuth 1.0a、ホスティングはRailway、ソース管理はGitHubとする。
- 既存のBasic認証、テスト基盤、Railway設定を再利用候補とする。
- 認証情報は環境変数等の安全な設定で扱い、コード、ログ、クライアントへ出力しない。
- Chrome拡張へAPIキーやBasic認証値を埋め込まない。Dashboard originは実行時だけ利用者が指定し、保存しない。拡張はブラウザーで事前に認証済みのDashboard originへだけ同期要求を送り、401時は認証情報を収集せず停止する。

## 対象外

- Yahoo!メールIMAP監視、メール本文からの商品抽出、メールID重複防止、Yahoo!メール認証画面
- JSONを中核とした保存方式
- 初期版の通知機能

これらは移行時の削除対象であり、今回の文書整備では実装を変更しない。
# Step 10 VR作品の安全除外

- VR作品は、女優・新作・sale・favorite_saleの取得、保存、候補生成、preview、投稿の全段階で対象外とする。
- 公式APIの構造化情報を優先し、明確なタイトル先頭マーカーだけを補助判定に用いる。価格の有無はこの判定に影響しない。

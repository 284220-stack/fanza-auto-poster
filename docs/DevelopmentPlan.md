# 開発計画

1 Stepを1機能として完結させる。各Step後に `npm run check`、`npm test`、`npm run build` を実行する。

| Step | 機能 | 完了条件 | テスト |
| --- | --- | --- | --- |
| 1 | PostgreSQL基盤 | `pg.Pool`、`DATABASE_URL`、接続確認、冪等な終了補助、SSL設定、マイグレーション用ディレクトリとコマンド | 未設定、成功、失敗、SSL、終了、接続文字列非露出 |
| 2 | 商品・投稿履歴モデル | 初期マイグレーションで商品、女優、関連、お気に入り、投稿履歴、設定のスキーマと制約・索引を作成 | マイグレーションの型検査、既存回帰 |
| 3A | 指定女優データ基盤 | 初期2名、Repository、Service、入力正規化と制約 | マイグレーション、Repository、Service |
| 3B | 指定女優管理API | 指定女優を操作する安全なHTTP API。DB未設定時もdashboardは起動可能 | API成功・入力不正・未検出・重複・削除競合・DB障害 |
| 3C | 指定女優管理画面 | 指定女優の追加・編集・検索・絞込み・有効化・無効化・削除を行う日本語UI | UI操作、aliases変換、APIエラー、読み込み状態、XSS対策 |
| 4A | 商品データ基盤 | productsのRepository・Service、入力検証、重複・存在確認 | Repository、URL・価格検証、重複、未検出 |
| 4B | 商品取得Provider基盤 | 共通モデル、正規化、Provider Registry、FakeProvider | 正規化、部分成功、Registry、取得条件 |
| 4C | FANZAセール取得Provider | 許可された方法によるセール候補取得Provider | 取得結果、正規化、異常時 |
| 4D | セール商品保存・更新 | Sale Provider結果をproductsへ作成・更新 | 作成、既存値維持、重複、部分失敗 |
| 4E | セール商品同期Runner | Providerと保存処理をDIし、同期結果と実行時間を集計 | 正常同期、Provider失敗、保存失敗、ログ |
| 5A | セール同期の実行基盤 | Runnerを手動API・一回実行CLIから安全に起動し、プロセス内多重起動を防止する | ロック、API、CLI終了コード、Pool終了、情報非露出 |
| 5B | 実環境スモークテスト基盤 | 実DB・DMM APIを安全に一回診断し、既定では保存しない | 設定、DB、Provider、check-only、persist、Pool終了 |
| 6A | 商品タイトル解析エンジン | タイトルの明示的な割引・キャンペーン事実を純粋関数で解析し、訴求候補を返す | 表記ゆれ、競合割引、作品名、候補順、警告 |
| 6B | キラーメッセージ生成基盤 | 解析済み事実だけから短い投稿冒頭候補をルール生成する | 優先順、複合候補、文字数、重複、誇張禁止 |
| 4F | 定期実行 | CronまたはRailwayスケジューラーからRunnerを起動 | 多重起動、再実行、失敗時 |
| 4D | お気に入り同期 | Chrome拡張と同期API、認証情報非保存 | 抽出、API認証、upsert |
| 5 | 商品取得アダプター | 状態、価格、セール/新作、動画、URLの正規化 | 正常、欠損、無効、販売終了 |
| 6 | 選定・条件判定 | 3区分、最大5、候補不足、重複防止 | 枠境界、30日、文面、除外 |
| 7 | 親投稿・返信 | 【PR】動画親投稿と親ID返信 | 文面、動画なし、返信先、DRY_RUN |
| 8 | 定期実行 | 重複起動防止、DB整合性、日本時間日次 | 多重起動、再実行、日付 |
| 9 | 管理画面・分析 | 設定、履歴、実績、同期、失敗理由 | API、Basic認証、UI |
| 10 | 旧機能撤去 | Yahoo!・JSON中心実装の削除とRailway確認 | 依存削除、回帰、デプロイ前検証 |

各Stepの共通完了条件は、設計文書との整合、対象テストの追加、上記3コマンドの成功、`git diff --check` の成功、機密情報を含めないことである。仕様変更時は実装前に設計文書を更新する。

Step 1では業務テーブルを作成しない。Step 2で初期スキーマを追加した。Repository、CRUD、商品取得、Chrome拡張、X投稿は後続Stepで実装する。
